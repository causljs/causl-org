Module @causl/checker

Node-side wrapper around the causl-check Rust binary distributed under tools/checker. The module exists to give JS/TS callers a typed, ergonomic entry point into the static rule engine: callers hand in a CauslModel IR, and a child process receives it on stdin, emits a JSON CheckerReport on stdout, and exits with status 0 (clean) or 1 (violations). Locating the binary is done by walking up from the package install root and consulting an env override, so the wrapper works in dev workspaces, in CI, and from a published package.

Property-based fuzz cannot prove the absence of a bug — it can only fail to find one. It also cannot explore unbounded state spaces, so long commit chains, large graphs, and rare resource interleavings can hide bugs that 1000 trials miss. This wrapper is the bridge to the bounded model checker that closes those gaps for projects within its configured size, depth, and resource bounds.

@causl/checker

npm wrapper around the causl-check Rust binary.

Status: v1 is a static IR linter — eight one-shot passes against the CauslModel IR. The bounded model checker described in SPEC §16.2 (state-space enumeration, async-resolution interleaving, Msg-dispatch traversal, conditional-branch enumeration, statechart conformance, replay determinism) is deferred to v1.x; I closed the capability-gap issue #272 as not-planned and parked the work under EPIC #282. Today's coverage table lives in docs/checker-coverage.md.

Usage

import { runChecker } from '@causl/checker'
import { createCausl } from '@causl/core'

const graph = createCausl()
// ...register nodes, run commits...
const model = graph.exportModel()
const report = await runChecker(model, { maxNodes: 100, maxCommits: 500 })
if (report.violations.length > 0) {
  console.error(report.violations)
  process.exit(1)
}

Or as a CLI:

causl-check < model.json
causl-check --input model.json --max-nodes 100

Install path

pnpm add -D @causl/checker

@causl/checker declares optionalDependencies on five per-platform packages — @causl/checker-linux-x64, @causl/checker-linux-arm64, @causl/checker-darwin-x64, @causl/checker-darwin-arm64, @causl/checker-win32-x64. Each one carries the prebuilt Rust binary for exactly its target, constrained by os/cpu fields in its package.json. npm and pnpm filter on those at install time, so a single platform host installs exactly one of the five and skips the other four (the esbuild / swc / biome shape).

The result: pnpm add -D @causl/checker on Linux, macOS, or Windows produces a working npx causl-check without rustc, without a postinstall network fetch, and without exposing the adopter's CI to corporate-proxy or air-gap failure modes.

The release pipeline that builds and publishes those five packages plus this wrapper is documented in docs/ci.md ("Release flow").

Binary resolution

The wrapper looks for the binary in this order:

  1. CAUSL_CHECK_BINARY env var.
  2. The matching @causl/checker-<target> optionalDependency's bin/ directory, resolved via require.resolve.
  3. tools/checker/target/release/causl-check (in-tree dev workspace).
  4. tools/checker/target/debug/causl-check.
  5. causl-check on $PATH.

For local development on a workspace checkout:

cargo build --release --manifest-path tools/checker/Cargo.toml
