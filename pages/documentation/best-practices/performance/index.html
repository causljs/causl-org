Performance Best Practices

Causl is fast enough for most application workloads out of the box: the benchmark suite anchors the engine at a 6.4× residual against MobX on the contract-bearing cells, decomposed as 1.84× engine baseline × 3.5× replay-determinism contract premium (see SPEC §17.5). The 3.5× premium is the price you pay for the four contract surfaces causl ships that MobX does not: commitLog rebuild, changedNodes set construction, GraphTime monotonicity, and readAt/snapshotAt retention. The 1.84× is the structural floor a JS engine that honours those contracts cannot cross.

This page covers the practical levers that move you toward that floor rather than away from it. None of them are exotic — they are habits. Pick them up early; they compound.

1. Exploit the equality-cutoff (Phase B / Phase C.5)

Causl's commit pipeline (SPEC §5.1) routes every staged input write through an Object.is comparison against the current value in Phase B. A write of the same value is treated as a no-op for the rest of the pipeline: lastWriteTime is not restamped in Phase C.5, Phase D will not re-enter any derived whose dep-set membership did not actually change, and Phase G subscribers never fire. The cost of "this input was already that value" is a single Object.is call.

The mechanical implication is that idempotent writes are free. You can lean on this aggressively in code paths that re-publish state from upstream sources (server pushes, form syncs, scroll position relays). Write whatever value you have; if it matches, causl swallows it. You do not need a hand-rolled "did this change?" guard at the call site.

// Fired on every scroll event; identical Y values short-circuit at Phase B.
window.addEventListener('scroll', () => {
  graph.commit(() => {
    graph.set(scrollY, window.scrollY);
  });
});

Where it stops being free is at object-valued inputs and derived outputs. Object.is is reference-comparison for objects, arrays, and Maps, so a freshly-rebuilt-but-deep-equal value will not short-circuit. Two patterns address this:

  1. Reuse references when nothing changed. In reducers and transforms, return the old reference rather than a new one when the structural diff is empty. Immer's produce does this by default; Redux Toolkit's reducer slices follow the same rule.
  2. Use a custom equality function in your derived. Project the value into a primitive (a hash, a sorted-key string, a version counter) and key downstream derivations off the projection. Phase B short-circuits, Phase D never re-enters.
Detailed explanation. Object.is is the one equality check the engine performs per write; full structural equality would change the contract surface (a structurally-equal-but- reference-different write would be observationally absent from commitLog, breaking replay determinism). The compromise is the one in the SPEC: Object.is at Phase B is mechanical and cheap, and adopters who want structural equality express it explicitly in the derived they care about. See SPEC §5.1 Phase B and Phase C.5 for the full mechanism.

2. Batch your writes — one commit, many sets

Every graph.commit() call runs phases A through H end to end: it builds a frozen Commit envelope, stamps a fresh GraphTime, intersects subscribers against changedNodes, refreshes commitLog (when commitHistoryCap > 0), and writes a per-commit input snapshot to the retention buffer. Most of that envelope cost is per-commit, not per-write. Ten writes inside one commit run one envelope; ten commits with one write each run ten.

      // Slow — ten envelopes.
for (const cell of cells) {
  graph.commit(() => { graph.set(cell.input, cell.value); });
}

// Fast — one envelope.
graph.commit(() => {
  for (const cell of cells) {
    graph.set(cell.input, cell.value);
  }
});

The rule of thumb: commit at the boundary of a user-intent unit. One keystroke, one tick, one frame, one network reply — those are commit boundaries. Looping over fields inside a single form submission is not; flush the whole form in one commit.

The React adapter does this for you on event handlers — anything you do inside an onClick or onChange handler is already wrapped in a single auto-batched commit. Where this falls down is async boundaries: a setTimeout callback or a Promise.then is its own commit unless you explicitly wrap it.

3. Avoid fan-out at commit boundaries

A "fan-out" in causl terms is a single input edge whose change cascades into a large transitively-affected derived set. The cost of a commit scales with the size of changedNodes: Phase D re-enters every dirty derived, Phase E adds them to the changedNodes set on the frozen Commit, and Phase G visits every subscriber whose dep-set intersects.

The fan-out the benchmark suite stresses is equality-cutoff-fanout-10k: one input feeding 10,000 derived nodes. Causl handles it — that is what the residual band contracts — but the cell sits at the high end of the residual because every one of those 10,000 derivations has to be visited at least to ask "did your dep-set change?".

The architectural lever is to collapse the fan-out into a pipeline. Instead of N derivations all reading the same input, route through one intermediate derived that does the work once and exposes a structurally smaller value downstream:

      // Anti-pattern: 10k cells each reading the same big input.
const sheet = input(graph, bigGrid);
const cells = grid.cells.map((c) =>
  derived(graph, g => evaluate(c, g.read(sheet)))
);

// Better: one summarising derived, downstream cells project from it.
const sheet = input(graph, bigGrid);
const evaluated = derived(graph, g => evaluateAll(g.read(sheet)));
const cells = grid.cells.map((c) =>
  derived(graph, g => g.read(evaluated)[c.id])
);

The pipeline form pays the evaluation cost once per commit and lets each leaf derived short-circuit on Object.is against the sub-projection. Where the anti-pattern visits 10,000 derived bodies per commit, the pipeline visits one + the leaves whose projected value actually changed.

Honest caveat. The pipeline form trades dep-graph depth for dep-graph breadth, and very deep linear chains have a separate cost (see SPEC §5.1 Amendment 3 — the iterative Phase D walker raised the depth ceiling past 12k on Node 22+, but registration is still O(depth)). The break-even is around fan-out of ~100 in our microbenches; below that the direct form wins on simplicity.

4. When to enable the WasmBackend

loadWasmBackend() from @causl/core/wasm returns a backend that runs the same engine in an opt-in WebAssembly substrate. As of the current ship, the WasmBackend is a TS-engine wrapper (Phase 1) — the full Rust port is deferred to post-0.9.0 behind GO/NO-GO criteria. The wrapper exists today because the contract surface and the loader stabilise the integration seam ahead of the engine swap.

The auto-adapt heuristic names the boundary adopters should care about. Enable WasmBackend when both of:

  • Graph size ≥ 50,000 nodes. Below this, the TS-engine wrapper's overhead is observable but the JS engine's absolute cost is still small enough that the swap is not worth the opt-in.
  • Median commit latency ≥ 1.0 ms. This is the floor where the swap starts to deliver a wall-clock win against application workloads. Below 1.0 ms, GC pressure and FFI envelope cost dominate; above it, the engine work amortises.

You measure both numbers with EngineTelemetry against a real workload — not synthetic benchmarks. The standard adoption pattern is the one in docs/wasm-adoption-guide.md:

async function tryWasmBackend() {

    return await loadWasmBackend();
  } catch (err) {
    if (err instanceof WasmBackendUnavailableError) return 'js';
    throw err;

const graph = createCausl({
  backend: await tryWasmBackend(),
  commitHistoryCap: 1000,
});
H1 hazard for adopters. Per the SPEC §15.1 amendment, graph.read(node) does not contractually return the same JavaScript reference across calls. Today's WasmBackend is a TS-engine wrapper so identity holds trivially; the day the real Rust serde/wasmgc bridges land, read() returns a fresh object per call as the value crosses the FFI boundary. Adopters who keyed React.memo or useMemo off the read return will silently re-render every commit. Key your memoisation on commit.time (the GraphTime exposed on Commit) or the per-node version counter exposed via EngineTelemetry instead.

# Gate: fail if any contract-bearing cell falls outside the §17.5 band.
pnpm --filter @causl/bench bench:gate

For day-to-day sub-second iteration, use pnpm bench with a profile flag — the tolerant-mode harness (run-tolerant.ts) aborts any cell that exceeds 2× its baseline so you get fast feedback on the change you just made without sitting through the full sweep.

The bench:check-hypotheses step consumes report/fair-fight-results.json and checks the median-band invariants declared in packages/bench/src/hypotheses/causl-hypotheses.ts. Those invariants are the SPEC §17.5 band, expressed as a property: mobx_median × 3.0 ≤ causl_median ≤ mobx_median × 8.0 on the three contract-bearing cells. A change that erodes the wave gains and pushes a cell past 8× fails the gate; a change that improves a cell past 3× also fails, because it has either shipped an architectural breakthrough (SPEC §17 amendment owed) or silently retired one of the four contract surfaces. Both directions force the conversation.

6. SPEC §17.5 as a target

The residual band is not a thing to be defeated; it is a structural property of the language plus the contract. Aim for it; do not aim past it. The four contracts MobX skips — commitLog, changedNodes, GraphTime monotonicity, readAt/snapshotAt retention — are exactly the features that make causl worth choosing for the workloads it targets. A change that pushes a contract-bearing cell sub-3× against MobX means one of those features has been silently retired. A change that pushes it past 8× means a perf seam the May-2026 wave closed has re-opened.

In day-to-day work, the band gives you a target you can aim for:

  • Your commit latency target is roughly MobX latency × 6.4 on the cell shape you most resemble. Below that, you are doing better than the post-wave anchor; above 8×, something is wrong with your graph shape.
  • Your fan-out target is whatever your application budget allows for the cell-count × commit-frequency product, not whatever the engine technically supports. The engine supports a lot; your frame budget does not.
  • Your batching target is one commit per user-intent unit. If you find yourself with a hot path that runs more than one commit per visible state transition, the next- cheapest perf win is almost always to merge them.

Where this stops mattering

Most applications never hit the boundary where any of this is load-bearing. A typical interactive UI has a few hundred nodes, a handful of commits per second, and a commit-budget measured in milliseconds of which causl spends tens of microseconds. The advice on this page exists for the workloads that do hit the boundary — large spreadsheets, dependency-graph editors, formula engines, anything where a single user gesture publishes thousands of writes. If you are not in that regime, optimise for readability and correctness first; the engine will keep up.
