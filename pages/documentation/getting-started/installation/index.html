Installation

Causl is published as a family of small packages under the @causl/* scope. You install only the pieces your application uses: the core engine alone is enough for a Node script or a non-React UI, and every other package — React bindings, async resource lifecycles, persistence, static checking, statecharts — is additive. This page covers install commands for all three major package managers, peer dependency requirements, TypeScript setup, the optional WebAssembly backend, and the per-package bundle budget.

Supported environments

Causl ships ESM-only modules targeting modern runtimes. The minimum supported versions are:

  • Node.js 22.0+ for server-side, SSR, and tooling use.
  • Any browser with ES2022 support — Chromium 102+, Firefox 102+, Safari 16+.
  • TypeScript 5.4+. The packages ship hand-written .d.ts declarations alongside the JavaScript.
  • React 18.3+ or React 19 for @causl/react (peer dependency).

Older Node and browser versions are not tested. There is no CommonJS build; if you need to require @causl/core from a CJS module, use the dynamic import() form or a small wrapper.

@causl/core

@causl/core is the semantic kernel — the dependency graph, the transactional commit, the deterministic recompute pipeline. Every other package depends on it. Start here; install nothing else until you need it.

      import { createCausl, input, derived } from '@causl/core';

graph.commit(() => {
  graph.set(count, 5);

@causl/core has zero runtime dependencies. The only declared peer dependency is fast-check, and it is marked optional — it is pulled in only by the @causl/core/testing subpath that exposes property-test helpers. You will never need to install it for production use.

@causl/sync

@causl/sync is the async resource lifecycle: fetchers, mutations, suspendable reads, and the race-row guarantees (S-1 stale-write, S-2 stale-read) that keep concurrent network calls from corrupting derived state. Install it when you have data that lives outside the graph — HTTP, IndexedDB, WebSocket, etc.

@causl/checker

@causl/checker is the static analyser: a CLI plus library entry point that walks your causl graph definitions and surfaces race rows (R-1 through R-8), cycle hazards, missing capability scopes, and unreachable derivations. It is a build-time tool — install it as a dev dependency.

# pnpm

The checker ships precompiled native binaries (@causl/checker-darwin-arm64, -linux-x64, -win32-x64, etc.) selected by the optional-deps mechanism — you do not install platform binaries directly. Run it with npx causl-check or wire it into a lint:graph npm script.

WasmBackend (optional)

Causl ships an opt-in WebAssembly backend entry point at @causl/core/wasm. It is bundled with @causl/core — no extra npm install step — and is loaded only if you call loadWasmBackend().

      import { createCausl } from '@causl/core';
import { loadWasmBackend, WasmBackendUnavailableError } from '@causl/core/wasm';

async function makeGraph() {
  try {
    const backend = await loadWasmBackend();
    return createCausl({ backend });
  } catch (err) {
    if (err instanceof WasmBackendUnavailableError) {
      // CSP rejected wasm-unsafe-eval, embedded runtime, etc.
      return createCausl();  // TS engine fallback
    }
    throw err;
  }
}
Current state. As of v0.9.0 the WASM backend is a TS-engine wrapper (Phase 1) — the public surface, the structured WasmBackendUnavailableError fallback chain, and the three-tier host compatibility matrix are all in place, but the underlying engine inside the WASM artefact is the TypeScript engine compiled through the bridge. The full Rust port is deferred to post-0.9.0 behind a GO/NO-GO review per SPEC §17.6. Adopters who turn on the WASM backend today get the contract; they get the throughput improvement once the Rust port lands. Cross-link: packages/core/wasm/README.md and docs/wasm-adoption-guide.md for the host-tier matrix, the bridge auto-detection, and the H1 reference-identity hazard that bites at the Rust-swap boundary (SPEC §15.1 amendment).

The three host tiers (wasmgc-builtins, wasmgc-classic, serde-json) are selected at module-load time by detectBridge(). You do not pin a tier in your package.json — the artefacts are lazily fetched, and only the tier your host actually supports is downloaded. If WebAssembly instantiation fails outright (CSP without 'wasm-unsafe-eval', embedded runtimes, etc.), the loader throws WasmBackendUnavailableError with a structured code field and you fall back to the TS engine at the construction seam.

      {
  "compilerOptions": {
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "Bundler",
    "strict": true,
    "esModuleInterop": true,
    "skipLibCheck": true,
    "jsx": "react-jsx",
    "lib": ["ES2022", "DOM", "DOM.Iterable"]
  }
}

Module resolution. Causl is ESM-only and uses package exports subpaths (@causl/core/wasm, @causl/core/testing, etc.). You need "moduleResolution": "Bundler" (or "NodeNext") so TypeScript reads the exports map. Older "moduleResolution": "Node" will silently miss the subpath types.

Strict mode. Causl's branded types (NodeId<T>, GraphTime) only catch the errors they are designed to catch under strict: true. A non-strict tsconfig erases the brand and a node from graph A passed to graph B will fail at runtime instead of at tsc. The team treats strict: true as a precondition; do not try to use causl without it.

JSX. If you use @causl/react, set "jsx": "react-jsx" (or "react-jsxdev" in dev). The classic "react" setting still works but is no longer recommended.

Bundle sizes

Causl publishes a size-limit CI gate on every published surface so the bundle budget is enforceable, not aspirational. The ceilings, per SPEC §14.2 and §17.6, are:

BundleCeiling (Brotli)Notes
@causl/core (full import) 20 KB Working target ~18 KB. Includes source-mapped error chains, devtools opt-in wiring (DCE'd in production), runtime IR validation in hydrate(), branded NodeId<T> / GraphTime, and the dehydrate() / hydrate() SSR pair.
@causl/core (createCausl-only) 15 KB The minimum import surface. Adopters who only register inputs and read derivations sit near this floor.
@causl/react 8 KB Absorbs the four §8.2 extension hooks plus the dev-mode "Update did not commit" warnings (DCE'd in production).
@causl/devtools-bridge (absent-extension path) 5 KB The bridge logic that the devtools: true opt-in in core now imports. The present-extension path is larger.
@causl/core/wasm entry stub 3 KB Loader skeleton (loadWasmBackend, detectBridge, WasmBackendUnavailableError, BridgeFeatures). Adopters who never import this entry pay zero bytes for it.
WASM Tier 1 (wasmgc-builtins) 60 KB Brotli ceiling Currently ~45 KB Brotli. Lazily fetched only on hosts that support wasm:js-string direct imports.
WASM Tier 2 (wasmgc-classic) 60 KB Brotli ceiling Currently ~50 KB Brotli. Lazily fetched on Safari 18.2+ and WasmGC hosts without the JS-string import binding.
WASM Tier 3 (serde-json) 80 KB Brotli ceiling (target) Universal-compat fallback. Lazily fetched only on hosts that hit Tier 3. See SPEC §17.6 for the current raw-byte divergence.

The CI gate size — bundle-size gate fails any PR that crosses these ceilings. The team holds the line on the budget: every kilobyte over 4.5 KB shipped to @causl/core required written team consensus per SPEC §14.2.
