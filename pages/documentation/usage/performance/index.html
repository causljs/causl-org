Performance

This page covers what causl promises about wall-clock speed, how to measure that promise against your own workload, the optimisations the engine already ships, and when to opt in to the WebAssembly backend. The short version is that causl pays a structural premium over mobx to ship replay-determinism, and that premium is bounded — not asymptotic — by a contract in SPEC §17.5.

If you are deciding whether causl is fast enough for your application, the honest answer is: read this page, run the bench harness against a workload that looks like yours, and consult the benchmarks page for the canonical seven-cell sweep. Synthetic microbenchmarks rarely predict app-level behaviour; the harness exists so you do not have to guess.

1. The capability-cost residual band

Causl is a transactional dependency-graph engine. mobx is a reactive observable engine. They look interchangeable from the outside — both track derivations, both fire subscribers when inputs change — but they ship different capabilities, and those capabilities cost cycles.

Specifically, causl ships four contract surfaces that mobx does not:

  • commitLog rebuilt on every commit (replay-deterministic IR).
  • changedNodes set construction for Phase G subscriber dispatch.
  • Monotonic GraphTime stamping per commit.
  • readAt / snapshotAt retention bookkeeping (historical reads against past commits).

SPEC §17.5 commits to a measurable, bounded cost for those surfaces. On three contract-bearing benchmark cells — equality-cutoff × 10000, equality-cutoff-fanout-10k × 10000, and spreadsheet-100x100 × 10000 — the engine guarantees:

mobx_median × 3.0  ≤  causl_median  ≤  mobx_median × 8.0

The band has two boundaries that mean different things:

  • Upper bound (8.0×) is the regression gate. Earlier in the project the equality-cutoff × 10000 cell ran at 1.94 ms vs mobx's 0.215 ms — about 9×, breaching the ceiling. A focused perf wave delivered a 24% drop (1.94 ms → 1.47 ms), pulling the residual to ~6.4× and giving the 8× ceiling about 25% headroom over the post-wave anchor. A future change that drives the cell back past 8× either reintroduces a per-write per-derived allocation seam or invents a new one; either way, the gate forces the conversation in review.
  • Lower bound (3.0×) is the capability-erosion gate. The ~0.9–1.1 ms causl pays beyond mobx on equality-cutoff is, from JavaScript, non-addressable: string-keyed Map probes, full-call Object.is, frozen Commit allocation, and the GC pressure those produce together set a structural floor. A future PR that delivers a sub-3× ratio on a contract-bearing cell has either shipped a real architectural breakthrough (the named candidate is promotion of the Rust-backed @causl/engine-wasm substrate to the default, per SPEC §17.6) or it has silently retired one of the four contract surfaces. The 3× floor flips that trade-off into a visible CI failure rather than a silent ship.

The residual decomposes as 1.84× engine baseline × 3.5× contract premium. The 1.84× factor is the only honest engine-on-engine microbench (op-tx-set-isolated-1k, with the hasDependents fast path that skips the staging/rollback round-trip on isolated inputs). The 3.5× factor is what the four contract surfaces above actually cost.

Methodology callout. Two cells in the wider engine-on-engine harness are excluded from the residual: op-tx-shadow-read-1k reads ~11× because mobx returns the new value directly inside runInAction without a shadow-Map probe, and op-commit-rollback-1k reads ~2.8× because mobx is not transactional — runInAction does not roll back on throw, so the comparator simulates a failed transaction with a second runInAction. Both are methodology asymmetries rooted in shipped capability, not inner-loop cost, and folding them into the residual would produce a number neither side could defend.

Post-Rust port projection

The numbers above describe the current TS engine. Once the Rust port lands (deferred behind GO/NO-GO criteria post-0.9.0), the projected residual band is 1.0× to 4.0× mobx_median on the same contract cells. The lower bound moves to parity because the structural floor that the JS engine cannot cross — Map allocation, Object.is dispatch, GC pressure on frozen Commit objects — is precisely what crossing the WASM/Rust boundary is allowed to bypass. The upper bound moves to 4× because the contract surfaces themselves do not get cheaper; they just get implemented in a substrate that walks pointers instead of hash maps.

Today's WasmBackend is a TS-engine wrapper (Phase 1 — see wasm-adoption-guide.md). The post-Rust band is a projection, not a promise.

2. Running the bench harness

The @causl/bench workspace ships every measurement that backs SPEC §17.5. You can run the same harness against your local checkout in two flavours: a quick run and a reportable run.

Quick run

pnpm bench

This drives packages/bench/src/run.ts with the default profile, sweeping the seven canonical cells (diamond, scrolling-viewport, dynamic-deps, async-race, batch-commit, equality-cutoff, large-fanout) at the production scales and printing wall-clock medians per library per cell. Use it in development to spot order-of-magnitude regressions in the inner loop you are working on.

Full report

pnpm bench:report

This drives src/report.ts, writes report/benchmark_results.json and report/benchmark_history.json, and emits a Markdown table with median Δ%, p95 Δ%, CoV, and per-cell win/loss attribution. The same JSON shape feeds the regression-gate (pnpm bench:gate) and the hypothesis-catalogue check (pnpm bench:check-hypotheses). Run this when you want a number to put in a PR body.

Each run aggregates a per-cell measurement across multiple trials of n=50 samples. Aggregate median / p95 / stddev are taken as the trial-medians, and CoV = stddev / median. A single n=50 stripe on the contract cells reads CoV in the 0.10–0.25 range, which is too wide to detect a 10% effect; aggregating five trials drops the trial-median's standard error far below the gate threshold. The published per-cell CoV envelopes — visible in every report and committed alongside regression-baseline.json — are the input you need to decide whether a delta you observe is signal or noise.

Detailed explanation. The harness honestly reports both wall-clock and stddev/CoV because the contract is about median, not best-case. If you find yourself wanting to quote a "fast run" number, that is the cue that your environment is noisier than CI's and you should run more trials, not pick the lowest sample. The CI gate intentionally fires on the median.

Targeted cells

For PR-sized investigations the focused driver runs only the three SPEC §17.5 cells in seconds rather than the 30+ minutes a full sweep takes:

pnpm --filter @causl/bench bench --scenarios equality-cutoff,equality-cutoff-fanout-10k,spreadsheet-100x100 --scale 10000

The full audit driver is packages/bench/src/freeze-impact-cli.ts; the smaller-cell entry points share the same harness so the numbers compose with the full report.

3. The equality-cutoff optimisation

Causl's most consequential commit-time optimisation is the equality cutoff: if a staged write to an input would set the same value already in the cell (Object.is(prev, next) returns true), the engine drops the write before Phase D and never recomputes downstream derived nodes.

This sounds obvious; the consequence is not. In a derived graph where a single input writes its dependent column on every commit, the equality cutoff turns a topological-recompute-of-everything into a topological-recompute-of-nothing. The equality-cutoff × 10000 cell is named after exactly this scenario: 10,000 derived nodes fanning out from one input that is written to with a value that compares equal to its current value.

import { createCausl, input, derived } from '@causl/core';

const graph = createCausl();
const source = input(graph, { count: 0 });
const doubled = derived(graph, (g) => g.read(source).count * 2);

graph.commit(() => {
  // Same value: equality cutoff drops the write, doubled is NOT recomputed.
  graph.set(source, { count: 0 });
});

Two things are worth knowing about this:

  1. The comparison is Object.is, not deep equality. graph.set(source, { count: 0 }) in the snippet above passes a new object reference each call, so Object.is returns false and the write commits. If you want the cutoff to fire for value-equal objects, you must keep the reference stable yourself, or compute the value into a variable and pass that variable in.
  2. The cutoff runs in Phase C (input-layer fast path), before Phase D (Kahn topo recompute). That is why equality-cutoff × 10000 is so much cheaper than a write that does propagate: when the cutoff fires, the engine never enters the topological walker. See SPEC §5.1 for the phase diagram.

If your application looks like the equality-cutoff cell — many writes that are no-ops at the value level — causl's wall-clock will track mobx's almost exactly modulo the 3.5× contract premium. If your application looks like scrolling-viewport — every commit mutates real data — the equality cutoff does not help, and you should profile.

4. Scrolling-viewport: a worked perf recovery

The scrolling-viewport × 10000 cell is a 1,000-cell virtualised grid scrolled at 60 Hz: every frame mounts roughly twenty new rows, unmounts twenty old rows, and writes per-row inputs. It is the cell most representative of "real DX" — mount/unmount churn under user input.

At an early-2026 snapshot, this cell was running at a catastrophic 654× mobx on the engine-on-engine harness. The root cause turned out to be a single hot helper, anyInputSubscriberIn, doing an O(n) walk over the subscriber list on every Phase G dispatch — and Phase G fires once per row mount. At a 1,000-row viewport, the walk cost was multiplied by the row count.

The recovery collapsed anyInputSubscriberIn to O(1) via the subscriptionsByNode index that was already maintained for the dispatch fast path. The hypothesis-catalogue row went from INVALIDATED — anyInputSubscriberIn at 26.92% (≥5%) to PASS — no match exceeds 5% threshold for "anyInputSubscriberIn"; the cell's median dropped by 97%, and the 654× reading collapsed into single-digit territory consistent with the §17.5 band. There is no longer a residual on this cell that the wave program needs to defend.

Why this is worth remembering. The 654× figure was not a competitor cherry-pick; it was the engine's own reported wall-clock against its own canonical workload. The recovery happened because the hypothesis catalogue named the suspect symbol, the bench harness measured the suspect symbol's contribution, and the fix was small enough to ship in one PR. If you observe a regression of similar shape in your own profile, the same loop applies: pnpm bench:check-hypotheses names the candidate symbols, pnpm bench:profile:cpu attributes the wall-clock, and the fix should land in one PR with a Perf delta in the body.

5. When to migrate to WasmBackend

Causl ships an opt-in WebAssembly backend at @causl/core/wasm. Today's implementation is a Phase-1 TS-engine wrapper: it gives you the ergonomics of the WASM loader and the host-tier compatibility shape, but the inner loop is still TypeScript. The full Rust port lands behind explicit GO/NO-GO criteria post-0.9.0.

The adapter heuristic lets adopters opt in to auto-adaption: causl probes the host, picks the highest available bridge tier (wasmgc-builtins > wasmgc-classic > serde-json), and falls back to the TS engine when no tier is available. The default is the TS engine; WasmBackend is opt-in even when the host supports it.

import { createCausl } from '@causl/core';
import { loadWasmBackend, WasmBackendUnavailableError } from '@causl/core/wasm';

async function chooseBackend() {
  try {
    return await loadWasmBackend();
  } catch (err) {
    if (err instanceof WasmBackendUnavailableError) return 'js';
    throw err;
  }
}

const graph = createCausl({ backend: await chooseBackend() });

The structured-code fallback chain (CAUSL_WASM_NOT_BUILT, CAUSL_WASM_UNAVAILABLE, CAUSL_WASM_CSP_BLOCKED, CAUSL_WASM_BRIDGE_UNAVAILABLE) means a single try/catch covers every host where WASM is not viable — CSP without 'wasm-unsafe-eval', embedded runtimes with WASM stripped, hosts where WebAssembly.Module is undefined.

You should migrate to WasmBackend when:

  1. Your workload sits on a contract-bearing cell shape. If pnpm bench:report against a workload that mirrors yours shows you near the 8× upper bound and the profile attributes the cost to engine inner-loop work (not user-provided derive functions), you are the population the Rust port is designed for. Once the native Rust phase lands, your residual moves toward the 1×–4× band; today's wrapper at least lets you stage the loader and adopter-side seams.
  2. You are willing to accept the H1 hazard. The day the real Rust bridges land, graph.read(node) on an object-valued node will return a fresh object per call as the value crosses the FFI boundary. Adopters who React.memo'd on the read reference, or used useMemo(() => transform(value), [value]) with value as the read return, will silently re-render every commit. SPEC §15.1 makes this explicit: reference identity is not contractual; key memoisation on commit.time or the per-node version counter EngineTelemetry surfaces.
  3. You measured the crossover. The bench:crossover-curve script in packages/bench/scripts/crossover-curve.ts sweeps the cell scale and prints the point at which WasmBackend overtakes the TS engine for your scenario shape. For most workloads under ~1,000 active derived nodes, the TS engine wins outright; the WASM substrate's per-call overhead exceeds its inner-loop advantage. Above ~10,000 active nodes the wrapper begins to break even, and the post-Rust projection puts the crossover much earlier.

You should not migrate to WasmBackend when your workload looks like the smallest cells. If "single mutation, single subscriber, no diamond, no async, no batch" describes your hot path, the TS engine is fast enough and the WASM loader is paying for a capability you do not use. Jotai will outpace any backend on that shape; causl is honest about the crossover.

6. Where the residual cannot be addressed

It is worth restating the §17.5 lower-bound argument in adopter terms. If your profile of a contract-bearing cell shows the bulk of the wall-clock in Map.prototype.get, Object.is, structured-clone of Commit deltas, or major-GC pauses, those are the floor: no JavaScript-level optimisation crosses below them while preserving the contract surfaces. The right lever is the WasmBackend, not a micro-optimisation in commit().

Conversely, if your profile shows the bulk of the wall-clock in user-provided derived bodies, the engine is already as fast as the contract allows; the answer is to look at the derive function itself. pnpm bench:profile:cpu causl <scenario> <scale> attributes user code separately from engine code in the flame graph, so the question of "is this me or is this causl?" is mechanically answerable.
