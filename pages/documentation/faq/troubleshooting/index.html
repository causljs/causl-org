FAQ — Troubleshooting

The errors and surprises in this page come from real bug reports, audit findings, and CI failures. Each entry names the symptom you see, the contract that explains it, and the fix — with citations into SPEC.md and the relevant PRs so you can verify the answer against the canonical source.

If your problem is not listed here, open an issue against iasbuilt/causl with the error class, the minimal reproduction, and your @causl/core version. Engine-level errors carry Error.cause with { commit, site, dependents } — include that payload verbatim.

Why does graph.read(node) return a different reference every commit?

Short answer: because reference identity across commits is not part of the contract. The TS engine happens to return the same reference today; the WASM backend will not. Memoise on commit.time or on the per-node version counter from EngineTelemetry — never on the return value of read().

Per SPEC §15.1, the contractual identity surface is value identity at a fixed GraphTime:

  • Object.is(graph.read(node)@t, graph.read(node)@t) holds for any two synchronous reads at the same GraphTime.
  • Object.is(graph.read(node)@t, graph.read(node)@t') may or may not hold across commits, and the answer depends on the backend.

Today's WasmBackend is a TS-engine wrapper, so an object-valued node returns the same reference trivially. The day the real Rust serde/wasmgc bridges land (deferred post-0.9.0), every read() will deserialise across the FFI boundary and return a fresh object per call. Code that memoised on the reference will silently re-render every commit after migrate('wasm').

This is the H1 hazard from the WASM adopter audit. The fix is mechanical:

      // WRONG — memoises on a reference the contract does not preserve.
const view = useMemo(() => transform(graph.read(profile)), [graph.read(profile)]);

// RIGHT — memoise on commit.time (the GraphTime the value belongs to).
const value = graph.read(profile);
const time  = graph.now();
const view  = useMemo(() => transform(value), [time]);
Detailed explanation. EngineTelemetry also exposes a per-node version counter that increments each commit a node changed. Use it when you need a key that only churns on relevant changes; commit.time churns on every commit, which is correct but coarser.

Why is my React component re-rendering every commit?

The most common cause is the same H1 hazard as above, pulled through useCauslNode or a useSyncExternalStore wrapper. A child component wraps itself in React.memo with the read value as the comparison key, expecting equal references to cut the render. Once you wire that against any backend that does not preserve identity across commits, the memo always misses.

  return <div>{profile.name}</div>;

  const profile = useCauslNode(graph, profileNode);
  const version = useCauslNodeVersion(graph, profileNode);
  return <Card profile={profile} key={version} />;
}

useCauslNode itself is fine — it subscribes via subscribeReads (SPEC §11.1) and only fires when the tracked read-set intersects the commit's changed set. The over-render is always downstream of the hook, at a memo boundary that keyed on a reference instead of on a value.

ComputeFailure during derive — what does it mean?

A ComputeFailure means a derivation's compute callback threw during Phase D (recompute fixpoint) of the commit pipeline. The error is wrapped — the original throw lives at error.cause along with the structured payload { commit, site, dependents } that names the in-flight commit, the derivation site, and the downstream derived nodes that depended on the failed one.

The pipeline catches the throw in Phase D's catch arm and rolls the §3 state back to byte-identical pre-commit values per Theorem 3 (Atomicity). No partial commit reaches Phase E. The whole transaction is reported as failed.

      import { createCausl, input, derived, ComputeFailure } from '@causl/core';

const graph    = createCausl();
const items    = input(graph, [] as readonly Item[]);
const totalUSD = derived(graph, g => {
  const list = g.read(items);
  // Bug: list[0] is undefined when items is empty.
  return list[0].price * list.length;
});

try {
  graph.commit(() => { /* nothing — first read of totalUSD also commits clean */ });
} catch (err) {
  if (err instanceof ComputeFailure) {
    console.error('site:', err.cause?.site);
    console.error('downstream:', err.cause?.dependents);
    console.error('original:', err.cause?.error);
  }
  throw err;
}

Fixes, in order of preference:

  1. Make the compute total — handle the empty/undefined case explicitly. Derivations are pure functions of their inputs; if some input value is "no value yet," that should be encoded in the type, not in a throw.
  2. If the error is genuinely exceptional, let it propagate. The commit fails atomically and the caller gets the audit trail. This is the design.
  3. If you need to preview whether a commit would throw without paying the publish cost, use graph.simulate(run) — it returns { status: 'failed', error } rather than throwing (SPEC §9).

NodeDisposedError on a node I'm sure exists

A NodeDisposedError fires on any read, subscribe, or tx.set against a node whose tombstone has been recorded by the engine. Two common causes:

  1. Use-after-dispose on a family-keyed node. Family-keyed nodes (statechart regions, per-row cells in a spreadsheet) tombstone when their parent scope disposes. Holding a stale NodeId from before the dispose and reading it after surfaces this error. Per SPEC §9.1 row 11, the tombstone lookup is exhaustive.
  2. Generational NodeId mismatch. Each NodeId now carries a generation tag. A node registered, disposed, and re-registered at the same index has a different generation; any stale handle from the first lifetime throws NodeDisposedError rather than silently aliasing the new node. This is the v1.x replacement for the ABA hazard the earlier non-generational design carried.

      // Wrong — holds the id across a family dispose.

familyDispose(graph, rows, 'r-42');
graph.read(row);              // throws NodeDisposedError
familyRegister(graph, rows, 'r-42');
graph.read(row);              // still throws — generation has bumped

// Right — re-derive the id after re-registration, or guard on isDisposed.
let row2 = familyRegister(graph, rows, 'r-42');
graph.read(row2);             // ok

The runtime guard is intentional. SPEC §13.4 considered promoting disposal to a compile-time discriminator (Node<T> = Live<T> | Disposed) and rejected it — it would have forced every consumer to handle the Disposed arm everywhere, a major API break for a hazard that a tombstone catches exhaustively at the call sites that matter.

CommitInProgressError in a subscriber

Subscribers fire in Phase G — after the frozen Commit has been published but before graph.commit() returns to the caller. Calling graph.commit() from inside a subscriber throws CommitInProgressError; the pipeline is single-writer by design (race-class row 7, commit-inside-subscribe reentry).

The two correct shapes:

      4

      // (a) Defer to a microtask if you genuinely need to chain a commit.
graph.subscribe(orders, () => {
  queueMicrotask(() => {
    graph.commit(() => graph.set(notified, true));
  });
});

// (b) Better: encode the dependency as a derived node so the engine
// handles it inside the same commit.
const notified = derived(graph, g => g.read(orders).length > 0);

Shape (b) is almost always the right answer. A subscriber-driven commit means the second commit's changedNodes is morally part of the first commit's effect, and the engine cannot give you the §3 Determinism guarantee across that boundary.

StaleTxError on tx.set

The tx object passed into commit(run) is only valid for the synchronous duration of run. Capturing it and using it later — typically by passing it into a promise then handler — throws StaleTxError on the deferred tx.set.

      10
    

      // WRONG — tx is captured and used after run returned.
graph.commit(tx => {
  fetch('/api/profile').then(res => {
    tx.set(name, res.name); // throws StaleTxError
  });
});

// RIGHT — fetch first, then commit.
const res = await fetch('/api/profile').then(r => r.json());
graph.commit(tx => { tx.set(name, res.name); });

If you have async resource state to manage, register it as a Resource via @causl/sync rather than reaching back into a stale transaction (SPEC §9.1 rows S-1 and S-2 are the race classes the adapter is designed to close).

CycleError thrown at first commit, not at registration

Per SPEC §9.1 Amendment 1, cycle detection moved from registration-time to first-commit-time. The earlier registration-time DFS was load-bearing on a chain of 1000 derivations (420 ms median against a 5 ms audit floor) and overflowed the V8 stack at 10000. Today's engine catches back-edges via a Phase-D post-recompute probe.

If your error trace looks like the cycle was introduced by a derivation closing over an outside-scope holder that mutated after registration, this is the row 8 race class. The probe will still catch it — but at the first commit that walks the SCC, not at the closing derived() call.

readAt returns the evicted arm

Every Behavior is undefined for t < registrationTime. A readAt(node, t) with t before the node existed surfaces the discriminated evicted arm with oldestRetainedTime: t_r — the earliest GraphTime where the read would succeed.

This is not a bug. The value was not "retained and dropped"; it was "never defined." SPEC §3 covers both the input and derived branches. The recovery breadcrumb is the field you should branch on:

if (result.status === 'evicted') {
  console.warn('earliest available:', result.oldestRetainedTime);
  return null;
}
return result.value;

WASM backend bundle-size warning in CI

The Tier 3 serde bridge currently ships at 213 KB raw / 66 KB Brotli. SPEC §17.6 sets the target ceiling at 200 KB raw / 80 KB Brotli. Brotli is comfortably under; raw is over by 13 KB.

This is a documented divergence, Option C — the size-limit cell in root package.json was ratcheted to 230 KB so the bundle-size CI gate absorbs the current measurement, and the SPEC row continues to document 200/80 as the long-run target. If your CI gate is reading the size-limit cell, it passes today; if you've written your own gate directly against the §17.6 row, you may need to relax it by up to ~30 KB raw until the gap closes.

When does the gap close? When the Rust engine port lands and wasm-opt is invoked directly with the feature flags wasm-pack rejects today. At that point the size-limit cell tightens back to ≤200 KB raw and the §17.6 current-state paragraph is removed.

Detailed explanation. Brotli is the wire size that actually matters for end users; raw is the bound that matters for CDN cold-start and for the bridge's claim to be small enough to inline. The SPEC keeps both, and the temporary divergence is on raw only.

Bench-gate is failing on a change that "shouldn't be slower"

The bench-gate is the bench-gate.yml CI workflow that compares your branch's median against the main-branch baseline. It gates on two thresholds:

  • COV_REGRESSION_DELTA = 0.08 — the coefficient-of-variation envelope. A 1.08× regression within the noise floor is treated as no regression. Anything outside the envelope is reported.
  • Per-suite absolute floor — sub-millisecond microbenchmarks (single-cell commits, equality cutoffs) have a per-benchmark absolute floor below which the percentage delta is suppressed entirely.

tinybench is configured for 200 ms warm-up, 1000 iterations, median reporting, with the noise floor computed from 30 runs of a no-op benchmark and reported alongside every result (docs/benchmark.md). If your change failed bench-gate, the report will name which benchmarks tripped which threshold.

Triage in order:

  1. Run the same suite locally with pnpm bench:gate -- --suite=<suite> against the baseline SHA. If the regression doesn't reproduce, it's noise — re-run the CI job.
  2. Check the CoV column in the report. A CoV > 0.05 on the failing benchmark means the noise floor is high for that suite on the runner; the 0.08 envelope may not be wide enough that day.
  3. If the regression reproduces locally and the CoV is tight (≤ 0.03), the regression is real. Find the change that caused it; do not raise the envelope.

Do not edit COV_REGRESSION_DELTA as a workaround. The 0.08 floor was calibrated against the 30-run no-op CoV; raising it to pass a flake teaches the gate to ignore real regressions on lower-noise runs.

The race-class checker is flagging code I know is safe

@causl/checker runs the static linter against the seventeen-row race-class catalogue from SPEC §9.1. The catalogue is reproduced and audited in docs/race-class-audit.md; the coverage table that maps each row to its detection layer (STATIC, PROPERTY, MODEL, RUNTIME-ONLY) is in docs/checker-coverage.md.

If the checker is flagging a pattern you believe is safe, walk this decision tree:

  1. Find the row. The diagnostic message names the row by number (e.g. "row 7 commit-inside-subscribe reentry"). Open docs/race-class-audit.md and read the row's intent.
  2. Check the layer. If the row is RUNTIME-ONLY, the linter should not be the one firing — file a bug. If it's STATIC, the linter is the contract surface, and "I know it's safe" usually means the analysis is conservative on a pattern that the runtime would still catch.
  3. Suppress narrowly. If the diagnostic is a documented conservative false positive, suppress with // causl-check-disable-next-line race-row-N and leave a one-line comment with the audit reasoning. Suppressing without naming the row defeats the auditability the catalogue is designed for.
  4. File a row. If the diagnostic represents a genuinely new pattern not covered by the seventeen rows, open a PR against the .github/PULL_REQUEST_TEMPLATE.md race-class anchor with the new row and its detection-layer assignment. Per SPEC commitment 12, every new §9.1 row ships with a layer assignment in the same PR.
Detailed explanation. The catalogue is intentionally over-eager on STATIC rows because the alternative — pushing detection to a property run that may not enumerate the offending interleaving — is strictly worse for adopter safety. The audit doc spells out which patterns are known-conservative; consult it before raising the priority of a suppression PR.

HydrationSchemaError on snapshot restore

@causl/persistence's hydrate() validates the restored snapshot against the engine's CauslModel schema (today schema: 3, see SPEC §4). A schema mismatch surfaces as HydrationSchemaError with the offending field and the schema version the engine expected vs. the version the snapshot carried.

Snapshot v1 (schema 1) and v2 (schema 2) have both been retired. Migration adapters live in @causl/persistence/migrate:

CapabilityViolation in a scoped subgraph

Scoped subgraphs (SPEC §16.2.1) limit which inputs a derivation can read and which outputs a subscriber can observe. A CapabilityViolation at runtime means a derivation tried to read a node outside its declared scope, or a subscriber tried to observe one. The error carries the offending NodeId and the violated capability name in cause.

The fix is either to widen the scope's capability set explicitly (if the read is intentional) or to factor the read out into a derivation inside the source scope that the destination scope can subscribe to (the design intent of the bridge registry). Casting your way around the check by reading through a captured graph reference works, but defeats the whole point of scoping — and the property suite will catch it under capability-leak.property.test.ts.
