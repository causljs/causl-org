FAQ — Comparing to Other Libraries

This page is the honest comparison. Causl is one point in a rich design space; pretending it dominates every other library would waste your time and ours. The state-management ecosystem in JavaScript is mature, opinionated, and full of libraries that solve real problems — MobX, Jotai, Redux, Recoil, Zustand, Valtio. Each one is the right answer for some application. Causl is the right answer for a narrower band — tangled dependency graphs where atomic commits, deterministic replay, and a frozen commit log are load-bearing — and it pays a measurable cost outside that band.

The structure of this page: a one-line summary table; one section per library covering what it does well, what causl does differently, and when to pick which; a section on the capability-cost band causl commits to in SPEC §17.5 commitment 13; and pointers to the per-library migration guides.

One-line summary

MobXDeep-proxy observables + transparent reactivityYou want zero-ceremony reactive objects and your team is fluent in proxies.
JotaiBottom-up atoms with derived atomsYou want React-first, atom-granular state without a graph identity story.
Redux (RTK)Single-store reducer + middleware ecosystemYou want time-travel devtools, RTK Query, and a huge middleware ecosystem.
RecoilAtoms + selectors with React-Suspense-first asyncYou want atoms with a Suspense-first story and you accept its experimental status.
ZustandTiny, hook-shaped store with manual selectorsYou want the smallest possible API surface and no graph at all.
ValtioProxy snapshots with structural sharingYou want MobX-style ergonomics without decorators or class observables.
CauslExplicit dependency graph + atomic commits + commit logYou have a tangled graph, you need atomic writes, and you want deterministic replay.
What "tangled" means. If you can model your state as a forest (slices that do not reach into each other), Redux/Zustand/Jotai are excellent. Causl earns its keep when nodes depend on other nodes in a real graph — spreadsheets, build systems, statecharts, planners, dependency-driven editors — and one user action mutates multiple inputs whose derived consumers must update together or not at all.

Causl vs MobX

MobX is the most direct philosophical cousin to causl. Both express derived values as functions of observed inputs; both auto-track dependencies; both believe the user should not have to maintain a dependency graph by hand. The differences are about boundaries — what counts as one observable change, and what the engine promises across that change.

What MobX does very well

  • Ergonomics. Deep-proxy observables make ordinary object mutation reactive. store.todos.push(t) notifies subscribers without ceremony. Causl requires you to model that push as tx.set(todos, [...current, t]) — correct and immutable, but more typing.
  • Speed on isolated writes. MobX's per-write path is shorter because there is no transaction envelope, no commit log, no monotonic GraphTime stamp. On the op-tx-set-isolated-1k microbench, causl runs at roughly 1.84× MobX's median — the engine baseline before any of causl's contract surfaces enter the picture.
  • Adoption. A decade of MobX patterns, mature React bindings (mobx-react-lite), and an enormous tutorial corpus.

What causl does differently

  • One commit, atomic-or-nothing. graph.commit(intent, tx => …) stages every write; if the body throws, the graph rolls back as if nothing happened. MobX's runInAction does not roll back on throw — partial mutations remain. For a spreadsheet, an order entry form, or any flow where partial state is worse than no state, this is the decisive difference.
  • Deterministic replay. Causl's commit log can be replayed on a fresh graph to reproduce byte-identical state. This underwrites the @causl/persistence snapshot/restore story and a future event-sourcing surface. MobX has no equivalent — observables capture present state, not the history that produced it.
  • One subscriber dispatch per commit. MobX's deep proxies generate one notification per field mutation. Causl's Phase G fires subscribers once at the end of a commit, with a frozen changedNodes set. The migration guide (docs/migration/from-mobx.md) calls out the consequence: observable({ a, b, c }) typically becomes three inputs, not one, because three fields changing is three changes.

Pick MobX when…

  • You are starting a new project, your state is a clean object tree, and you do not need replay or atomic commits.
  • You are writing a deeply nested editor (e.g. a form with dozens of fields) and the verbosity of tx.set per field would dominate the actual work.
  • You need to ship in days, not weeks, and the team already knows MobX.

Pick causl when…

  • One user action mutates multiple inputs that must commit together or not at all.
  • You need a commit log — for undo/redo, audit, time-travel, or persistence.
  • You need replay determinism: a recorded commit stream must produce identical state on every engine instance.
The capability-cost band. SPEC §17.5 commitment 13 commits causl to the band mobx_median × 3.0 ≤ causl_median ≤ mobx_median × 8.0 on contract-bearing benchmark cells (equality-cutoff × 10000, equality-cutoff-fanout-10k × 10000, spreadsheet-100x100 × 10000). The post-wave anchor is 6.4×, decomposed as 1.84× engine baseline × 3.5× replay-determinism contract premium. The 3.5× is what causl pays for the four contract surfaces MobX does not ship: commitLog rebuild on every commit, changedNodes set construction, GraphTime monotonicity stamping, and readAt/snapshotAt retention bookkeeping. If you do not need those, MobX is faster — honestly so. If you do need them, you are buying the 3.5× on purpose.

The migration guide at docs/migration/from-mobx.md covers the mechanical translation table and the structural decisions (record-shaped observables, autorun vs subscribe, why runInAction is a deceptive equivalent for commit).

Causl vs Jotai

Jotai's bottom-up atom model is the closest in API shape to causl's node model. Both have a "primitive" atom/input and a "derived" atom/derived. Both compose by reading dependencies in a body function. The differences are about where the graph lives and how lifecycle works.

What Jotai does very well

  • React-first ergonomics. Atoms are module-level constants; useAtom is the only hook you need most of the time. The mental model maps cleanly onto React's component tree.
  • Atom families. atomFamily is the canonical pattern for per-entity state and Jotai's implementation is mature.
  • Bundle size. Smaller than causl — no commit log, no transaction envelope, no GraphTime.

What causl does differently

  • The graph is an object you hold. createCausl() returns a graph handle; nodes are registered on it. Jotai's atoms are global by default and per-Provider-scoped on demand. Causl's pattern makes multi-instance ergonomics (server-side rendering, tabs, test fixtures) the default rather than an opt-in.
  • Atomic commits across atoms. Jotai sets one atom at a time; setting three atoms in a row is three notifications. Causl's graph.commit stages all writes and fires subscribers once.
  • Static lint and race detection. @causl/checker is a Rust-backed static IR linter that catches dependency-graph hazards at lint time. Jotai has no equivalent — the discipline is review-based.

Pick Jotai when…

  • Your app is React-first and you want atom-granular state without a graph identity story.
  • You do not need atomic multi-write commits — one atom per user action is the norm.
  • You want the smallest possible reactive primitive and you are happy with module-global state.

Pick causl when…

  • You need atomic writes across many nodes per user action.
  • You need a commit log for replay, persistence, or undo/redo.
  • You want static checking of dependency-graph hazards.

The migration guide at docs/migration/from-jotai.md covers the mechanical translation (atom → graph.input, derived atoms → graph.derived) and the structural decisions (atom families, persistence, async atoms).

Causl vs Redux (and Redux Toolkit)

Redux is the dominant single-store reducer library and Redux Toolkit (RTK) is the modern shape: createSlice, createAsyncThunk, RTK Query. Causl's commit log is philosophically close to Redux's action log — both treat the history of changes as a first-class artifact. The differences are about shape (graph vs single object) and derived values (memoized selectors vs an actual dependency graph).

What Redux does very well

  • Devtools. The Redux DevTools time-travel debugger is one of the best developer experiences in front-end engineering. Causl's commit log can be inspected, but the Redux DevTools surface remains unmatched.
  • Middleware ecosystem. A decade of middlewares for logging, persistence, sagas, observables, undo, and RTK Query for server state.
  • Single source of truth. One store, one reducer tree. Easy to reason about.

What causl does differently

  • Derived values are nodes, not selectors. Redux's reselect memoizes selector functions; the dependency graph is implicit in the selector composition. Causl's graph.derived registers the node in an actual graph and recomputes via Kahn topological sort in Phase D. For a graph with many overlapping selectors, this is dramatically more efficient and removes the "did I memoize this correctly" footgun.
  • Messages, not actions. Causl's React surface uses typed messages ({ kind, …fields }) and a typed useDispatch<Msg>(). Redux's loose { type, payload } is exhaustively pattern-matched in causl via createUpdate.
  • No middleware layer. Causl handles cross-cutting concerns through graph.subscribeCommits (commit-level observers). For async I/O, @causl/sync models resources directly rather than through thunks or sagas.

Pick Redux/RTK when…

  • You want time-travel devtools and the broader Redux ecosystem.
  • Your state is well-modeled as a tree of slices with no cross-slice derived values.
  • You depend on RTK Query for server-state caching and a migration would be expensive.

Pick causl when…

  • You have cross-slice derived values and your reselect graph is getting tangled.
  • You want typed messages and exhaustive pattern matching on action shape.
  • You want the commit log to be a structural commitment of the engine, not a middleware add-on.

The migration guide at docs/migration/from-redux.md covers the mapping from slices to inputs + derived nodes, the move from thunks to @causl/sync resources, and the discipline of value-bearing messages.

Causl vs Recoil

Recoil is Meta's atom/selector library, originally positioned as the "graph-of-state" answer for React. Its design influenced both Jotai and causl. As of 2026 Recoil is effectively unmaintained — the public guidance has been to migrate to Jotai — so causl is rarely a head-to-head competitor today. The comparison is still useful if you are reading old patterns.

What Recoil aimed for

  • Atoms and selectors as first-class concepts.
  • Suspense-first async selectors.
  • Concurrent-mode-friendly snapshots.

What causl does differently

  • The graph is explicit. Recoil's selectors compose into an implicit graph; causl's nodes are registered on a handle and the topology is observable.
  • Commits, not snapshots. Recoil's snapshot/transaction story was always partial. Causl's commit is atomic-or-nothing by construction.
  • Active development. Causl is shipping; Recoil is in maintenance mode.

Pick Recoil when…

If you have an existing Recoil codebase that works and a migration is not on the roadmap, stay. If you are starting fresh, the practical alternatives in 2026 are Jotai (closest shape) or causl (if you need the contract surfaces).

Causl vs Zustand

Zustand is the smallest store in widespread use — one hook, a state object, a set function. It is deliberately, beautifully minimal. Causl is none of those things.

What Zustand does very well

  • API surface. Three concepts — create, the state object, and the set function. You can teach the whole library in five minutes.
  • Bundle size. Tiny.
  • No graph. If you do not have a dependency graph problem, you do not need a dependency graph library.

What causl does differently

Causl has a substantially larger API surface (the canonical seven Graph methods plus a curated second tier, per SPEC §12), an actual dependency graph, atomic commits, a commit log, and persistence/sync packages. If you do not need any of that, Zustand is the better tool. Honest.

Pick Zustand when…

  • Your state is small and your derived values are computed inline in React components.
  • You want the absolute minimum API surface.
  • You do not have a dependency graph problem.

Pick causl when…

  • You started with Zustand, your state grew, and you are now juggling useMemo-based derived selectors that re-compute too often or too little.
  • You realize "derived value X depends on derived values Y and Z" is a graph you no longer want to maintain by hand.

Causl vs Valtio

Valtio is "MobX without the decorators" — proxy-based reactivity over a plain object, with a snapshot function for React. The trade-off space is very close to MobX's.

What Valtio does very well

  • Ergonomics. proxy({ count: 0 }) and then state.count++ — reactivity is invisible.
  • React story. useSnapshot gives a structurally-shared snapshot for stable references.
  • No build step. Pure proxies, no decorators or compilers.

What causl does differently

The same differences as MobX, structurally: atomic commits, deterministic replay, an explicit graph, a static checker. Valtio's atomic story is subscribe with debouncing; causl's is a transactional engine.

Pick Valtio when…

  • You want MobX's ergonomics without decorators.
  • Your state is a clean object tree and your derived values are simple.

Pick causl when…

  • You need the contract surfaces — commit log, atomicity-on-throw, replay determinism — that no proxy library ships.

The capability-cost band, restated

It would be dishonest to end this page without naming the cost. Causl's SPEC §17.5 commitment 13 binds the engine to a measurable band against MobX on three contract-bearing benchmark cells:

mobx_median × 3.0  ≤  causl_median  ≤  mobx_median × 8.0

The post-wave anchor is 6.4×, decomposed as 1.84× engine baseline × 3.5× replay-determinism contract premium. The 1.84× is the honest engine-on-engine cost — what causl pays purely to be causl, before any contract surface enters the picture. The 3.5× is what causl pays for the four contract surfaces MobX does not ship:

  • commitLog rebuild on every commit (the ring buffer that underwrites replay).
  • changedNodes set construction for subscribers.
  • GraphTime monotonicity stamping (the now tick that orders every commit).
  • readAt/snapshotAt retention bookkeeping (the MVCC seam for time-travel reads).

The upper bound (8.0×) is a regression gate. A PR that drives a contract-bearing cell past 8× either reintroduces a per-write allocation seam the May-2026 perf wave closed or invents a new one. The gate forces the conversation.

The lower bound (3.0×) is a capability-erosion gate. A PR that delivers a sub-3× ratio has either (a) shipped a real architectural breakthrough — promotion of the @causl/engine-wasm substrate to the default JS engine is the named candidate (SPEC §17.6) — or (b) silently retired one of the four contract surfaces by trading capability for speed. The 3× floor flips that trade-off into a visible failure rather than a silent ship.

If you do not need replay determinism and a commit log, you are paying for surfaces you will not use. MobX is the right answer for that workload; we will say so on this page until the contract surfaces become free, which they will not, because the laws of JavaScript GC do not bend.

A working example, three ways

Same problem in three libraries to make the differences concrete: a counter and its doubled derivation.

MobX:

import { observable, computed, runInAction } from 'mobx'

const store = observable({ count: 0 })
const doubled = computed(() => store.count * 2)

runInAction(() => { store.count = 5 })
console.log(doubled.get())  // 10

Jotai:

import { atom, getDefaultStore } from 'jotai'

const countAtom = atom(0)
const doubledAtom = atom((get) => get(countAtom) * 2)

const store = getDefaultStore()
store.set(countAtom, 5)
console.log(store.get(doubledAtom))  // 10

Causl:

import { createCausl } from '@causl/core'

const graph = createCausl()
const count = graph.input('count', 0)
const doubled = graph.derived('doubled', (get) => get(count) * 2)

graph.commit('set-count', (tx) => { tx.set(count, 5) })
console.log(graph.read(doubled))  // 10

For the counter alone, MobX and Jotai are smaller and faster. Causl's shape pays off when the next user action has to set count, label, and color at once — one commit, one subscriber dispatch, one frozen Commit in the log, atomic-or-nothing if anything throws.
